# 极简操作模型

## 设计目标

操作模型服务一句话：

```text
看一遍，就剪完。
```

这意味着操作必须少、快、明确。用户不能在播放中频繁停下来找按钮，也不能在复杂菜单里配置参数。

第一版应该优先支持“边看边做决定”：

```text
要不要这一段。
看哪里。
是否切开。
是否锁住。
是否推近或拉远。
是否保存。
```

## 操作原则

```text
高频动作必须在手柄快捷按钮上。
中心视野尽量只显示视频、取景框和必要状态。
复杂设置放到低频工作台，不打断播放。
动作要能单手完成。
每个动作都要立刻给视觉反馈。
```

## 核心动作

### Cut

含义：

```text
在当前时间点形成一个剪辑边界。
```

用户感受：

```text
这里断一下。
从这里开始是新的镜头判断。
```

协议语义：

```text
当前 ViewPathPoint 写入 cut=true。
后端不跨 cut 做平滑插值。
```

### Keep

含义：

```text
当前片段进入保留状态。
```

用户感受：

```text
这段有用，继续出现在最终视频里。
```

协议语义：

```text
写入 enabled=true。
```

### Drop

含义：

```text
当前片段进入丢弃状态。
```

用户感受：

```text
这段没用，快速跳过或不进入成片。
```

协议语义：

```text
写入 enabled=false。
后端导出时跳过对应区间。
```

体验建议：

```text
进入 Drop 时可以自动提高播放倍速。
退出 Drop 后回到之前倍速。
倍速只影响观看体验，不改变 ViewPathPoint.tMs。
```

### Lock

含义：

```text
锁定当前取景方向。
```

用户感受：

```text
这个角度很好，别再跟着我的头动了。
```

协议语义：

```text
写入 locked=true。
后续点可以保持或平滑延续当前 center。
```

### FOV

含义：

```text
调整虚拟镜头视野。
```

用户感受：

```text
推近一点。
拉远一点。
```

协议语义：

```text
写入 fov.h / fov.v。
输出比例仍固定为 16:9，FOV 只改变取景范围。
```

### Save

含义：

```text
保存当前路径 patch。
```

用户感受：

```text
这一段剪辑判断已经记下来了。
```

协议语义：

```text
提交 ViewPathPatch。
通过 replaceRange 覆盖对应时间段。
```

## 推荐手柄映射

第一版可以按“右手主操作、左手辅助”的思路设计。

```text
Trigger 按住：
取景框跟随 controller ray 或 head gaze。

Trigger 松开：
锁定当前取景方向。

A / joystick press：
打开快捷环。

快捷环扇区：
Cut / Keep / Drop / Lock / Save。

右摇杆上下：
调 FOV。

右摇杆左右：
切换播放倍速。

B：
取消当前快捷环或退出临时状态。
```

## 快捷环

快捷环适合高频动作，因为用户不需要看一堆面板。

交互：

```text
按住打开。
拖向扇区。
扇区高亮。
松开确认。
拖回中心取消。
```

第一版扇区：

```text
Cut
Keep
Drop
Lock / Unlock
Save
```

## 为什么不是复杂时间线

这个项目不是要把桌面剪辑软件搬进 VR。

桌面时间线适合精修，但不适合黑客松第一阶段要表达的核心爽点。这里要先证明：

```text
360 视频的粗剪和取景，可以在观看过程中完成。
```

所以第一版不追求：

```text
多轨道。
复杂字幕。
音频混音。
精确逐帧调参。
大量插件面板。
```

第一版追求：

```text
快速看完。
快速取景。
快速标记。
快速导出。
```

## 关键体验指标

```text
用户不暂停也能完成大部分操作。
每个高频动作不超过一次按住/松开。
播放结束时已经产生可用路径。
用户能清楚知道当前是 Keep、Drop 还是 Lock。
误操作可以通过 replaceRange 重新覆盖。
```
