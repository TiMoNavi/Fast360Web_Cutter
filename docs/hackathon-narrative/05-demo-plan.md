# Demo 方案

## 总原则

```text
实机视频优先。
PC 截图补可信度。
二维码补可试用感。
现场实时操作只作为加分项，不作为主链路。
```

原因：

```text
黑客松现场网络、Quest 投屏、WebXR 权限和浏览器状态都可能不稳定。
主叙事不能押在实时设备一定成功上。
```

## 主演示视频

准备一段 40-60 秒视频，内容顺序固定：

```text
1. 进入 360 素材。
2. 看到 16:9 取景框。
3. Look：看向重点并记录。
4. Lock：稳定看向主体。
5. Skip：跳过无效片段。
6. Cut：形成镜头边界。
7. FOV：推近或拉远。
8. 出现 ViewPath / 注意力路径示意。
9. 展示导出 16:9 MP4。
```

视频建议画面：

```text
主画面：头显投屏或 WebXR 视角。
角落：真人戴头显或手柄操作画面。
结尾：360 原片与 16:9 导出对比。
```

## PC 端截图

PPT 第 5 页和第 7 页放 PC 端截图，证明当前项目有真实页面。

截图优先级：

```text
1. PC editor / XR session 页面：能看到 360 播放、取景框或控制区。
2. 移动端视频页：能看到视频、session、导出入口或二维码。
3. 导出结果页：能看到 MP4 下载或 export 状态。
```

截图旁边只写短说明：

```text
MVP 已跑通：上传、session、路径提交、后端 smoke render、下载。
```

## 二维码

二维码只承担“可以试”的补充，不承担主叙事。

二维码指向规则：

```text
优先：现场 LAN 或部署环境中的真实试用入口。
可选：移动端视频页、XR session 页、PC editor 页、备用公开视频页。
禁止：已删除的 /hackathon-deck。
```

PPT 上二维码旁边写：

```text
现场试用入口 / 备用演示入口
```

不要写：

```text
完整商业版体验
一键分享到所有平台
生产级长视频导出
```

## 备用方案

如果 Quest / WebXR / 网络 / 投屏失败：

```text
1. 直接播放提前录好的实机视频。
2. 用 PC 截图解释当前页面和后端闭环。
3. 用 ViewPath 图解释技术边界。
4. 用导出结果视频证明输出可行。
```

备用讲法：

```text
现场 WebXR 设备链路有不确定性，所以我把主交互录成实机视频。
这段视频展示的是目标体验；当前工程闭环则由 PC 页面和后端导出截图证明。
```

## 素材清单

```text
demo-quest-webxr.mp4
Quest / WebXR 实机交互视频。

pc-editor-session.png
PC 端 session 或 editor 截图。

mobile-entry-or-qr.png
移动端入口或二维码截图。

viewpath-diagram.png
头显动作 -> ViewPath -> 后端导出的技术图。

source-to-export.mp4
360 原片到 16:9 成片的对比视频。
```

在重新制作 PPT 前，先按 `00-visual-material-board.md` 把真实截图、录屏、输出视频和二维码补齐。不要再用纯文字页占位。

## 现场摆放

PPT 第 5 页建议布局：

```text
左侧 70%：实机视频。
右上 20%：PC 端截图。
右下 10%：二维码。
底部一句话：视频优先保证稳定，二维码用于现场试用。
```

## 验收标准

```text
不依赖现场 Quest 成功也能讲完整。
评委能看到真实 WebXR 交互画面。
评委能看到 PC 页面和后端导出不是纯概念。
二维码不会指向已删除页面。
```
