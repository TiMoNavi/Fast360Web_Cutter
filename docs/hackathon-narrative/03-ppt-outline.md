# PPT 大纲

## 设计原则

```text
8 页以内。
每页只表达一个判断。
主画面优先放实机视频、PC 截图、流程图和二维码。
不要把 PPT 做成商业计划书。
不要堆数据库、API 和长列表。
```

## 第 1 页：标题

标题：

```text
看一遍，就能发
```

副标题：

```text
一个把 360 素材观看过程变成注意力路径的 WebXR 空间导演层
```

主视觉：

```text
左侧：Quest / WebXR 实机视频截图。
右侧：导出的普通 16:9 视频截图。
```

讲解目的：

```text
先让评委记住一句话和一个画面。
```

## 第 2 页：痛点

标题：

```text
360 相机拍到全部，但很多素材卡在发布前
```

页面文案：

```text
拍摄时不用对准。
发布时必须重新决定观众看哪里。
很多素材不是没有内容，而是没有走完取景、导出、分享这一步。
```

主视觉：

```text
拍到全部 -> 看过素材 -> 取景决策太重 -> 没发出去
```

讲解目的：

```text
把问题从“剪辑功能”转成“发布门槛”。
```

## 第 3 页：定位

标题：

```text
不是另一个 360 剪辑器，而是最轻空间导演层
```

页面文案：

```text
成熟工具负责精修。
这个原型负责第一次观看时生成取景意图。
```

主视觉：

```text
360 相机 -> 空间导演层 -> 后端/成熟工具 -> 内容平台
```

讲解目的：

```text
对影石赞助商友好，避免被直接拿去和成熟产品比较。
```

## 第 4 页：流程

标题：

```text
上传 -> 观看 -> 注意力路径 -> 导出 -> 分享
```

页面文案：

```text
用户不是重新剪一遍，而是在观看中顺手做取景决策。
```

主视觉：

```text
Web 上传
WebXR 观看
ViewPath / 注意力路径
Backend 16:9 导出
Download / Share
```

讲解目的：

```text
解释完整闭环，但不陷入工程细节。
```

## 第 5 页：Demo

标题：

```text
实机视频优先：看、锁、切、跳过、导出
```

页面文案：

```text
主画面播放 Quest / WebXR 录屏。
旁边放 PC 端截图和现场二维码。
```

主视觉：

```text
70% 屏幕：实机视频。
20% 屏幕：PC editor 或移动端流程截图。
10% 屏幕：二维码。
```

二维码说明：

```text
指向现场可访问的真实试用入口或备用演示页。
不要指向已删除的 /hackathon-deck。
```

讲解目的：

```text
保证现场稳定，同时保留“可以试”的可信度。
```

## 第 6 页：技术边界

标题：

```text
不是录屏，是 ViewPath / 注意力路径
```

页面文案：

```text
WebXR 记录取景意图。
后端按路径重新投影。
路径可以覆盖、重放、重新渲染，也适合 AI 先生成、人再修正。
```

主视觉：

```text
头显动作 -> ViewPath 点列 -> 后端 remap -> 16:9 MP4
```

讲解目的：

```text
让技术评委相信这不是一次性视频捕获。
```

## 第 7 页：现实完成度

标题：

```text
学生黑客松 MVP：已跑通闭环，真实头显体验继续补齐
```

页面文案：

```text
已验证：上传、session、路径提交、后端 smoke render、下载。
建设中：真实头显采样、controller 快捷操作、取景框、5Hz sampler。
下一步：生产级分片渲染队列和分享出口。
```

主视觉：

```text
已跑通 / 建设中 / 下一步 三栏。
```

讲解目的：

```text
诚实说明现实情况，避免过度承诺。
```

## 第 8 页：收尾

标题：

```text
相机捕捉全部，我的系统快速确定注意力
```

页面文案：

```text
从观看 360，到分享成片。
看一遍，就能发。
```

主视觉：

```text
360 原片 + 注意力路径 + 16:9 成片。
```

讲解目的：

```text
用一句话收束，让评委记住项目位置。
```
