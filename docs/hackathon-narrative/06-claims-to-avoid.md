# 不要这样讲

这份文档用来防止叙事跑偏。项目可以酷，可以大胆，但不要把它讲成另一个东西。

## 不要讲成传统剪辑软件

避免说：

```text
这是一个 VR 版 Premiere。
这是一个完整视频剪辑软件。
我们要在 VR 里实现专业剪辑台。
```

更适合说：

```text
这是一个 VR 原生的 360 取景剪辑方法。
第一阶段聚焦边看边剪、快速取景和路径生成。
```

原因：

```text
传统剪辑软件的功能太大，会稀释项目最酷的点。
这个项目的核心不是多轨道编辑，而是空间取景决策。
```

## 不要讲成纯 AI 自动剪辑

避免说：

```text
AI 自动帮你剪完。
用户不用做任何判断。
```

更适合说：

```text
人类在 VR 里做观看和取景判断，系统把这些判断变成可渲染路径。
AI 未来可以辅助平滑、推荐和补全，但不替代核心意图。
```

原因：

```text
这个项目最有辨识度的是空间交互，不是自动化口号。
```

## 不要硬扯具体商业生态

避免说：

```text
这是某个相机品牌的官方工作流补丁。
这是某个生态必须需要的插件。
```

更适合说：

```text
它天然适合 360 相机素材。
它想证明一种新的 VR 剪辑方式。
```

原因：

```text
硬扯商业会让项目显得功利，而且容易把“酷炫的新剪辑方法”讲窄。
```

## 不要把未完成能力说成已经完成

当前可以诚实说：

```text
MVP 已经跑通上传、session、路径提交、后端 smoke render 和下载。
```

不要说：

```text
真实头显采样已经完整可用。
生产级长视频队列已经完成。
播放完一定能立刻生成最终生产级成片。
```

更准确说：

```text
目标体验是播放完即得到完整剪辑路径。
当前后端已有 smoke render，生产级分片队列是下一步。
```

## 不要把 WebXR 讲成最终渲染器

避免说：

```text
WebXR 直接输出最终 MP4。
VR 页面负责生成成片。
```

更适合说：

```text
WebXR 负责记录剪辑意图。
Backend 负责正式渲染。
Web 负责素材入口和结果出口。
```

原因：

```text
这是项目最重要的工程边界。
如果讲错，会让系统听起来像一次性录屏工具，而不是可回放的剪辑管线。
```

## 不要过度承诺“无需等待”

可以说：

```text
用户不需要在剪辑决策上长久等待。
观看结束时，剪辑路径已经形成。
```

谨慎说：

```text
正式 MP4 导出仍需要后端渲染时间。
目标是让剪辑等待变少，而不是宣称所有视频都瞬间完成生产级导出。
```

原因：

```text
长视频 360 重投影本身有计算成本。
更稳的表达是：边播边剪，播放完就有路径；导出可在后端继续优化。
```

## 不要让概念变复杂

避免现场展开太多：

```text
完整数据库设计。
所有 API 字段。
复杂模块边界。
各种未来扩展。
```

优先讲：

```text
为什么用 VR 剪 360。
用户怎么边看边剪。
为什么不是录屏而是路径。
播放完为什么能进入导出。
```

一句提醒：

```text
黑客松现场先让人觉得“这个方式对”，再补技术可信度。
```
