# 核心故事

## 一句话

```text
这是一个 VR 原生的 360 视频剪辑工具：你进入视频空间，看一遍，边看边剪，播放结束时剪辑路径也完成了。
```

更短一点：

```text
看一遍，就剪完。
```

更有画面感一点：

```text
它像一个戴在 VR 里的虚拟取景器，让你直接在 360 视频里决定最终观众应该看哪里。
```

## 30 秒版本

360 相机把整个空间都拍下来了，但我们后期处理时，通常还是坐在 2D 屏幕前，在时间线上反复拖动、打关键帧、调 yaw、pitch 和 FOV。

我觉得这件事不自然。

360 视频本来就是空间素材，VR 本来就是空间观看方式，所以剪 360 视频最自然的方法，应该是在 VR 里直接剪。

这个项目就是一个 WebXR 轻量剪辑原型：用户进入 360 视频球幕，一边播放一边用头显、手柄、取景框和快捷按钮做剪辑决策。系统不录屏，而是把这些决策压缩成一条可回放的取景路径。视频看完时，路径已经完成，后端就可以按这条路径导出普通 16:9 成片。

## 3 分钟版本

### 1. 问题不是“怎么播放 360 视频”

360 视频已经很容易播放了，真正麻烦的是出片。

拍摄时，360 相机记录了所有方向；发布时，大多数观众看的还是普通平面视频。于是创作者必须决定：

```text
这一秒看前方。
下一秒转到左边。
这里推近。
这里切掉。
这里锁住主体。
```

传统方法是在平面剪辑软件里打关键帧。问题是，创作者面对的是参数，不是空间。你要一边看一个被压扁的 360 画面，一边猜最终视角。

### 2. 这个项目的判断

我的判断是：

```text
VR 和 360 视频是天然适合的。
```

不是因为 VR 更炫，而是因为它把创作者放回了正确的位置：视频空间里面。

当创作者站在 360 视频中心，剪辑不再是填写参数，而是非常直接的动作：

```text
我看向哪里，哪里就是候选画面。
我按下保留，这段就留下。
我按下丢弃，这段就跳过。
我锁定取景框，镜头就稳定跟住。
我调 FOV，就是推近或拉远。
```

### 3. WebXR 让它变轻

这个项目不是做一个重型 VR 桌面剪辑软件。它的入口应该很轻：

```text
普通 Web 上传素材。
Quest 或支持 WebXR 的浏览器打开链接。
进入 360 视频空间。
看一遍，边看边做剪辑动作。
回到 Web 下载结果。
```

WebXR 的价值在这里不是“浏览器也能跑 VR”这么简单，而是让这个体验变成一个轻量入口，不需要先安装复杂软件，不需要先学习专业剪辑界面。

### 4. 它不是录屏

这个项目最重要的技术边界是：WebXR 不直接生成最终视频。

WebXR 做的是记录剪辑意图：

```text
时间点。
视角中心。
FOV。
保留或丢弃。
切段边界。
锁定状态。
```

这些内容形成 ViewPathPatch 和 ViewPathPoint。后端再根据这条路径正式渲染出 MP4。

这意味着剪辑行为可以被回放、覆盖、修正，也可以在后端用稳定算法导出，而不是依赖一次性的屏幕录制。

### 5. 最终想达到的感觉

理想状态下，用户不需要“剪很久”。

他只需要完整看一遍素材。在观看过程中，他已经顺手完成了剪辑判断。播放结束时，系统已经知道：

```text
哪些片段要。
哪些片段不要。
每一段应该看哪里。
什么时候切。
什么时候推近。
什么时候锁住。
```

这就是这个项目想表达的东西：

```text
看一遍，就剪完。
```
