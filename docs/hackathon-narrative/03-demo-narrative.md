# Demo 叙事

## 演示主线

Demo 不要讲成“我实现了很多模块”。更好的主线是：

```text
我用 VR 把 360 视频剪了一遍。
```

演示顺序可以非常简单：

```text
上传一个 360 视频。
打开 WebXR 链接。
进入 360 球幕。
播放视频。
边看边用快捷按钮做剪辑。
播放结束。
生成一条取景路径。
后端导出普通 16:9 视频。
```

## 开场说法

可以这样开场：

```text
360 相机最厉害的地方，是它把整个现场都拍下来了。
但问题也在这里：最后发出去的视频只能有一个画面。

传统做法是在 2D 时间线上慢慢调视角。
我的想法是，既然素材本来就是 360 度的，为什么不直接进到这个空间里剪？

所以这个项目的目标很简单：看一遍，就剪完。
```

## 第一步：普通 Web 是入口

讲法：

```text
普通 Web 只负责素材入口和结果出口。
这里上传 360 视频，创建一个剪辑 session，然后得到一个 WebXR 入口。
```

不要在这里展开太多后台逻辑。这个页面只是把用户送进剪辑空间。

可以强调：

```text
手机或浏览器负责管理素材。
真正的剪辑发生在 VR 里。
```

## 第二步：进入 WebXR 空间

讲法：

```text
进入之后，用户不是在看一个播放器窗口，而是站在 360 视频里面。
中心视野是视频本身，取景框代表最终输出画面。
```

这里的视觉重点应该是：

```text
球幕视频。
取景框。
少量状态。
尽量少的 UI。
```

## 第三步：边播边剪

这是 Demo 的核心。

讲法：

```text
视频继续播放，我不用停下来打关键帧。
我看向某个方向，取景框跟过去。
我按一下，就保留当前视角。
我遇到无用片段，就进入 Drop。
我想要稳定镜头，就 Lock。
我想推近，就调 FOV。
```

要强调“动作很少，但语义足够”：

```text
Cut：这里形成一个边界。
Keep：这段保留。
Drop：这段跳过。
Lock：这一段视角锁住。
FOV：推近或拉远。
Save：保存当前路径。
```

## 第四步：不是录屏，是路径

讲法：

```text
这里不是在录屏。
WebXR 端记录的是剪辑意图：什么时候看哪里、哪些片段保留、哪里切开、FOV 怎么变化。
这些内容会变成一条低频 ViewPath。
```

这句话很重要，因为它把项目从“VR 播放器录屏”提升到“可回放的剪辑方法”。

可以补一句：

```text
路径可以覆盖、重放、重新渲染，所以它比一次性录屏更像真正的剪辑。
```

## 第五步：播放完，路径完成

讲法：

```text
理想状态下，我不需要看完之后再剪一遍。
看视频的过程就是剪辑过程。
播放结束时，系统已经有了一条完整的取景路径。
```

这里可以把 slogan 放出来：

```text
看一遍，就剪完。
```

## 第六步：后端导出

讲法：

```text
最后后端按这条路径把 360 视频重新投影成普通 16:9 MP4。
WebXR 不负责生成最终视频，它负责记录人类的空间剪辑意图。
```

如果要说明当前状态，可以诚实地说：

```text
当前 MVP 已经跑通上传、session、路径提交、后端 smoke render 和下载。
真正的头显采样、快捷按钮和生产级分片队列，是下一阶段要补齐的完整体验。
```

## 现场节奏

3 分钟版本：

```text
30 秒：痛点和核心想法。
60 秒：演示进入 WebXR 和球幕播放。
60 秒：演示边看边剪的几个动作。
30 秒：解释路径和后端导出。
```

5 分钟版本：

```text
45 秒：为什么 VR 适合剪 360。
60 秒：普通 Web 上传和入口。
120 秒：WebXR 里边看边剪。
45 秒：路径协议和后端导出。
30 秒：当前完成度和下一步。
```
